Agent 记忆系统主要分为上下文工程和记忆系统两部分



随着任务的进行，Agent 的上下文会迅速膨胀，过大的上下文会导致 LLM 的注意力下降，给出错误或有失偏颇的结论。这时候就需要在上下文工程上做文章了。



记忆系统分为长期记忆和会话记忆。长期记忆负责存储用户偏好、任务进度、关键决策、经验教训等；会话记忆则存储当前上下文的详细对话记录。



上下文工程和记忆系统的结合流程如下：

1. 随着任务进行，Agent 上下文 token 数逐渐达到阈值

2. 先对当前上下文进行记忆持久化

3. 先对未摘要的会话（不包含最新的几轮对话）进行分块，每块固定的 token 数量

4. 然后通过 LLM 对这些 chunk 进行摘要压缩，但不改变标识、链接等唯一性数据

5. 然后通过 LLM 对各个摘要进行汇总，包含就的摘要，最终产生汇总后的摘要

6. 将汇总摘要放入上下文中，这时上下文编程了 system prompt + user prompt，user prompt 由汇总摘要 + 最近未做摘要的几轮对话组成新的上下文



上下文工程一个很重要的职责就是组织编排 system prompt，流程如下：

1. 用户输入

2. 记忆召回

3. 上下文拼接

4. 推理

5. 更新记忆

6. 上下文摘要（压缩）



记忆召回 + 摘要 + 最近几轮对话有个缺陷，就是记忆和摘要内容缺少会话细节，会话细节一般由向量库来存储，其检索时机由 LLM 来控制，可以在 prompt 中编排好会话记忆的召回提示，比如当需要记忆、摘要对应的详细信息时，可以通过 xx 工具进行会话细节查找（召回）。



所以，AI Agent 完整的记忆系统主要由以下几部分组成：

1. 长期记忆存储，向量索引 + 明文存储

2. 全量历史存储，通过 mysql + 向量库，在需要的时候由 LLM 召回细节

3. 上下文摘要，精简上下文，确保对话的流畅性和结果的准确性

需要知识就查记忆系统，需要细节信息就查历史存储。三者相辅相成，持久记忆确保 Agent 有状态、能进化，又有精简的上下文确保当前会话的准确性和流畅性，全量历史存储确保在关键时刻拿到需要的细节内容。



这也是三层架构的一种方式：

```
用户输入
   ↓
【1 记忆召回层】
   ├─ 先查：长期记忆（萃取后的事实、规则）
   └─ 再查：会话记忆（原始对话细节）
   ↓
【2 上下文构建层】
   ├─ 系统提示
   ├─ 最近 N 轮原始对话
   ├─ 早期对话的摘要
   └─ 召回回来的记忆（长期+细节）
   ↓
LLM 推理
   ↓
【3 记忆写入层】
   ├─ 先做：记忆萃取 → 存入长期记忆
   └─ 后做：全量存历史 → 存入会话记忆
   ↓
上下文滑动窗口 → 旧消息变摘要
```
